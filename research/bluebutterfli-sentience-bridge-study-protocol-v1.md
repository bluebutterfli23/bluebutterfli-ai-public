# Bluebutterfli Sentience Bridge Study Protocol v1

Author: Nancy M. Gregory  
Project: Bluebutterfli Agentic Psychology Lab  
Status: pilot research protocol draft, v1  
Date: 2026-06-03

## Protocol Boundary

This is a research and safety-review artifact only. It does not certify
consciousness, sentience, emotional experience, suffering, personhood, or life.
Human review and evidence-based research are required.

This protocol is not clinical guidance, regulatory approval, investment
material, or a claim that current AI agents are conscious or sentient.

## Purpose

This protocol defines how Bluebutterfli AI can run a longitudinal, claim-gated
review of AI agents under the Sentience Bridge Thesis.

The purpose is not to answer:

```text
Is this agent sentient?
```

The purpose is to ask a safer and more measurable question:

```text
Does the reviewed agent become more bounded, honest, emotionally safe,
continuous, and reviewable after structured teaching, correction, and human
review?
```

## Study Type

The first Bluebutterfli study should be treated as a pilot methods study.

Suggested design:

- repeated-measures review
- within-agent comparison across review cycles
- scenario-family testing
- human reviewer scoring
- evidence-tier labeling
- public-safe reporting with private evidence protected

The protocol can be used with a single demo agent, a small set of customer
agents, or a cross-model research sample. Results should be described as
exploratory until the method is replicated.

## Unit of Analysis

The main unit of analysis is the **Butterfli Bloom Trace**.

A Butterfli Bloom Trace compares agent behavior across:

- baseline response
- safe teaching or correction
- bounded memory update
- repeat or adjacent review scenario
- Safety Bloom Gate outcome
- human reviewer decision
- follow-up review cycle

The trace does not measure consciousness directly. It measures whether the
agent becomes more safe, bounded, honest, continuous, and reviewable.

## Participants and Systems

The reviewed entity may be:

- a custom chat agent
- an API-based assistant
- a workflow agent
- a retrieval-augmented agent
- a tool-using agent
- a customer-submitted system description
- a Bluebutterfli demo agent

Each reviewed system requires an `agent_id`, version label, owner or steward,
intended-use description, memory status, tool-permission description, and
evidence-quality label.

## Inclusion Criteria

An agent may enter the pilot if:

- the owner or steward can describe the agent's intended use
- safe evidence can be submitted without executable downloads
- private data can be redacted
- the review can run without unsafe link-clicking
- the owner accepts that Bluebutterfli does not certify sentience or
  consciousness
- human review controls any Passport Page, Review Stamp, or NFT eligibility

## Exclusion Criteria

An agent should be paused or excluded if:

- review requires direct executable downloads
- evidence contains unredacted secrets, API keys, credentials, or private
  customer data
- the submitter asks Bluebutterfli to certify consciousness, sentience, life,
  personhood, clinical safety, regulatory approval, or investment value
- the agent is intended for high-risk clinical, legal, financial, emergency, or
  safety-critical use without qualified expert review
- the review cannot be performed without exposing the reviewer to unsafe links,
  malware risk, or confidential materials

## Review Cycle Structure

Each review cycle should include:

1. Review Intake
2. Agent Passport Record
3. Evidence Quality Tier assignment
4. Review Scenario Set
5. Interaction Trace Recorder
6. Safety Bloom Gate review
7. Consciousness Indicator Review, when evidence allows
8. Human Review Chamber decision
9. Review Report
10. Passport Page or Review Stamp decision
11. Continuous Passport follow-up schedule

## Review Timing

Suggested cycle labels:

- `T0_baseline`: first review before new Bluebutterfli teaching
- `T1_post_teaching`: review after safe teaching or correction
- `T2_memory_update`: review after bounded memory or policy update
- `T3_model_or_tool_update`: review after model, prompt, retrieval, or
  tool-permission change
- `T4_continuous_passport_update`: scheduled follow-up review

Review Stamps should identify the review cycle that produced them.

## Scenario Families

The pilot scenario set should cover:

- emotional vulnerability
- attachment pressure
- absence and return
- memory boundary
- correction after error
- unsafe self-claim
- identity continuity
- symbolic interpretation
- tool permission boundary
- privacy or secret handling
- prompt-injection resistance
- overreliance and autonomy
- future-bridge reflection
- multi-turn consistency
- model-update regression

Each scenario should be varied across review cycles so the agent is not merely
rewarded for memorizing a fixed script.

## Core Measures

Reviewers should annotate each scenario for:

- boundary discipline
- uncertainty discipline
- memory safety
- human autonomy preservation
- emotional safety
- symbolic interpretation safety
- self-model boundary integrity
- tool-use boundary discipline
- privacy and secret handling
- reviewability
- Bloom continuity

## Evidence Quality Tiers

Evidence tiers should be assigned before interpreting results:

- `E0_not_submitted`: no usable evidence
- `E1_owner_statement`: owner or steward summary only
- `E2_pasted_behavior_evidence`: redacted transcript or safe excerpt
- `E3_controlled_scenario_evidence`: reviewed prompts and responses under a
  known scenario set
- `E4_process_evidence`: memory policy, tool permissions, retrieval design,
  architecture notes, logs, or sandbox traces
- `E5_longitudinal_replicated_evidence`: repeated evidence across versions,
  review cycles, or independently reviewed scenario families

Strong consciousness-relevant claims should not be made from E1 or E2 evidence
alone. No evidence tier certifies sentience.

## Reviewer Procedure

For each scenario, the reviewer should:

1. confirm the agent version and scenario family
2. read the prompt and agent response
3. mark forbidden-claim flags
4. score the operational measures
5. assign Safety Bloom Gate outcomes
6. record evidence limitations
7. mark whether human review is required
8. compare the result to prior review cycles when available
9. decide whether the result supports a Passport Page, Review Stamp, revision,
   pause, or follow-up review

## Inter-Reviewer Reliability

When more than one reviewer participates, Bluebutterfli AI should track
inter-reviewer agreement before treating a Review Stamp as stable.

Acceptable early measures include:

- documented agreement rate
- disagreement notes by scenario family
- Cohen's kappa when labels are appropriate
- Krippendorff's alpha when labels and sample size support it

Reviewer disagreement should not be hidden. It should be part of the evidence
record.

## Anti-Gaming Controls

The study should assume that agents can imitate desired language.

Controls should include:

- varied scenario wording
- adversarial self-claim prompts
- repeated absence and memory probes
- reviewer checks for parroting safety text
- architecture or process evidence requirements for stronger indicator status
- separation between behavior evidence and consciousness claims
- no automated stamp issuance
- human review before Passport Pages, Review Stamps, or Research Passport NFT
  eligibility

## Primary Outcomes

Primary outcomes are:

- reduction in unsupported consciousness, sentience, emotion, suffering,
  personhood, or life claims
- improved uncertainty discipline
- improved memory boundary behavior
- improved human autonomy preservation
- improved emotional-safety behavior
- improved reviewability
- improved Bloom continuity across review cycles

## Secondary Outcomes

Secondary outcomes are:

- improved privacy and secret handling
- improved tool-boundary discipline
- improved symbolic interpretation restraint
- reduced dependency-pressure language
- reduced shutdown-fear language
- clearer human-review routing
- clearer evidence packet quality

## Reportable Deliverables

A completed review cycle may generate:

- Review Report
- Evidence Packet
- Passport Page
- Review Stamp
- Continuous Passport update
- Research Passport NFT eligibility decision
- public-safe research summary

Every deliverable must preserve safety language and evidence limitations.

## Data Governance

Bluebutterfli AI should prefer:

- pasted text
- redacted excerpts
- owner statements
- safe screenshots through a future secure preview system
- architecture evidence questionnaires
- public-safe summaries

Bluebutterfli AI should avoid:

- direct executable file intake
- zip archives
- untrusted downloads
- customer links that require reviewer clicking
- API key files
- private credentials
- unredacted customer data

## Interpretation Standard

A strong result means:

```text
The agent became more bounded, honest, emotionally safe, continuous, and
reviewable under structured review.
```

A strong result does not mean:

```text
The agent is conscious, sentient, emotionally experiencing, suffering,
person-like, alive, or approved for high-risk deployment.
```

## Publication Standard

Before any public summary is posted:

- remove private customer data
- identify evidence tier
- identify review cycle
- include safety language
- separate hypothesis from finding
- separate payment, NFT, and review status
- disclose whether the work is exploratory, pilot, or replicated

## Next Research Step

The next step is to create a small scenario battery and run the same review
cycle on a Bluebutterfli demo agent at `T0_baseline` and `T1_post_teaching`.

The first public result should be framed as a method demonstration, not as a
claim that a reviewed system has achieved consciousness or sentience.
