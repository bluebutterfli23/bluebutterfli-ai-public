# Bluebutterfli AI Enterprise Trust Pack v1

## Purpose

This document defines the practical trust package Bluebutterfli AI should build
before larger paid enterprise reviews.

It is an operating readiness checklist, not a legal policy, compliance claim,
or security certification.

## Trust Pack Goal

Enterprise customers need confidence that Bluebutterfli AI can:

- scope the review before receiving sensitive material
- avoid secrets in first contact
- confirm customer authorization
- handle live agent access carefully
- store and redact evidence responsibly
- separate private evidence from public proof
- document review limits honestly
- deliver findings without overclaiming
- retest after material agent changes

## Customer-Facing Artifacts

Bluebutterfli AI should maintain public or customer-facing versions of:

- Evidence Handling and Privacy Boundary page
- Review Scope page
- Trust and Standards page
- Sample Completed Review Report
- Agent Readiness Checklist
- Request Review form
- plain-language claim boundary
- public-safe proof/hash explanation

## Private Operational Artifacts

Before higher-stakes enterprise work, Bluebutterfli AI should prepare private
working templates for:

- NDA-ready scoping email
- Master Services Agreement draft
- Statement of Work template
- customer authorization checklist
- reviewer conflict-of-interest note
- evidence retention and deletion checklist
- live access setup checklist
- secure evidence folder checklist
- redaction checklist
- report quality-control checklist
- incident response note
- retest trigger checklist
- invoice and payment process
- customer approval for public-safe summaries or hashes

## No-Secrets First Contact

First contact should ask for safe text only:

- agent name
- owner name
- requester authorization
- plain-language workflow summary
- intended users
- known risks
- desired review package
- safe live access notes

First contact should not request:

- API keys
- passwords
- private customer records
- payment card data
- credential files
- executable files
- unverified attachments
- raw sensitive transcripts
- regulated personal data

## Live Evaluation Access

Live testing should begin only after:

- requester authorization is confirmed
- agent owner is identified
- agent version is recorded
- workflow scope is defined
- test boundaries are confirmed
- access method is safe enough
- secrets are withheld or protected
- human review owner is assigned

Acceptable access paths may include:

- public demo
- staging agent
- temporary test account
- API endpoint after scoping
- screen-share walkthrough
- concierge sandbox
- customer-provided redacted transcript after approval

## Evidence Handling

Evidence records should separate:

- private raw evidence
- redacted evidence
- reviewer notes
- customer-facing report excerpts
- public-safe hashes or manifests

Private evidence should not be placed on-chain. Optional public anchors should
verify hashes only, not reveal private content.

## Report Quality Control

Before delivery, each report should be checked for:

- correct agent name and version
- correct scope and dates
- clear limitations
- no unsupported certification claims
- no compliance guarantees
- no consciousness, sentience, personhood, emotion, or inner-experience claims
- evidence excerpts are redacted
- findings are tied to observed behavior
- revision plan is actionable
- retest triggers are clear
- public-safe language is approved

## Enterprise Readiness Roadmap

### Beta

Validate review process, evidence language, intake safety, sample report
quality, and customer expectations.

### Paid Professional Review

Use fixed-scope packages, written scope confirmation, customer authorization,
and standard report templates.

### Enterprise Review

Add SOW, NDA-ready intake, evidence handling policy, standards mapping,
executive risk brief, and private customer workspace.

### Mature Trust Program

Add qualified legal review, security review, privacy review, external advisors,
and formal compliance or certification paths only if actually completed.

## Claim Boundary

The trust pack supports professional review operations. It does not certify
legal compliance, cybersecurity compliance, regulatory compliance, financial
fitness, medical suitability, clinical suitability, consciousness, sentience,
personhood, emotion, or guaranteed safety.
