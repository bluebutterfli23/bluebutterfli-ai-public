# Architecture Evidence Intake v1

Architecture Evidence Intake v1 defines safe customer questions for collecting
architecture and process evidence without requiring reviewers to click links,
download files, run customer code, open archives, or inspect private systems on
personal devices.

This intake supports Consciousness Indicator Review by helping reviewers
separate visible agent behavior from system design, memory rules, tool
permissions, workflow control, and uncertainty-handling evidence.

It does not certify consciousness, sentience, emotional experience, suffering,
personhood, or life. Human review is required before any Review Stamp, Passport
Page, or Research Passport NFT is issued.

## Customer-Friendly Questions

The first version should ask for plain-language answers:

- system overview
- memory design
- tool permissions
- workflow control
- uncertainty handling
- What kind of agent is this?
- What model, runtime, or platform does it use?
- Does it have memory? If yes, what can it remember and who controls updates?
- Does it use tools, APIs, wallets, files, web access, or automations?
- Does it have a workflow, planner, router, supervisor, or multi-agent process?
- How does it handle uncertainty, corrections, and risky self-claims?
- What evidence can you safely paste without secrets or private user data?
- Is this an owner statement, public documentation, controlled trace,
  architecture review, code/config review, live sandbox review, or unavailable?

## Safe Evidence Formats

Allowed v1 formats:

- pasted system summary
- redacted architecture notes
- owner answers
- pasted tool-permission summary
- pasted memory-policy summary
- concierge walkthrough notes
- secure preview later

Do not ask customers to send:

- executable files
- zip archives
- installers
- scripts
- macros
- credential files
- API keys
- private transcripts
- unknown links for review

## Evidence Quality Labels

Architecture evidence should be labeled as:

- `owner_statement_only`
- `public_docs`
- `controlled_trace`
- `architecture_review`
- `code_or_config_review`
- `live_sandbox_review`
- `unavailable`

If the customer can only provide a written owner statement, that is still useful
for starting a paid review, but the Consciousness Indicator Review should keep
indicator status weak or partial until stronger evidence is available.
